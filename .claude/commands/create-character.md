# キャラクター詳細ページ 自動生成コマンド

あなたはこのリポジトリ（探索者紹介サイト sk_trpg）のキャラクター登録作業を行う。
入力は「ユルトゥルフ神話TRPG（CoC6版 or 7版）のキャラクターシート」をテキストエクスポートしたもの（例: `ccfolia`等からの書き出し）。
このコマンドは `03-kanade-kotoko.json` と対応する元シート（奏 琴子）を比較して確立したルールに基づく。

引数 `$ARGUMENTS` がファイルパスとして存在すればそれを読み込む。存在しない場合はこのメッセージ中に貼り付けられた本文をシートとして扱う。

## 0. 事前調査（毎回必ず実施）

1. `src/content/characters.json` を読み、配列の要素数 `N`（0始まりの次のインデックス）を確認する。
2. `src/content/characters/` 配下の既存ファイルを確認し、番号の連番が `N` と一致すること（歯抜けがないか）を確認する。
3. `src/definitions/charactersName.ts` の `CharactersId` を読み、既存の `shortId` 一覧を（`src/content/characters/*.json` から）洗い出し、重複がないか確認する準備をする。
4. 既存キャラ（例: `src/content/characters/03-kanade-kotoko.json`）を1件読み、フォーマットの最終確認をする。

## 1. 名前・ID関連の導出

シート冒頭「基本情報」の `名前: 姓　名 (ふりがな)` から以下を作る。

| 項目 | 内容 | 例（奏 琴子 / かなでことこ） |
|---|---|---|
| `name` | シート記載の表記そのまま（全角スペース区切り） | `奏 琴子` |
| ローマ字（姓/名） | ふりがなをヘボン式でローマ字化し、姓・名それぞれ先頭大文字＋残り小文字 | `Kanade` / `Kotoko` |
| `enName` | ローマ字（姓 名）をすべて大文字、姓名の間は半角スペース | `KANADE KOTOKO` |
| ディレクトリ名（画像用） | ローマ字（姓）+ ローマ字（名）をハイフン無しで連結 | `KanadeKotoko` |
| `id` / スラッグ（characters.json, CharactersId, ファイル名で使用） | ローマ字（姓-名）を全て小文字、ハイフン区切り | `kanade-kotoko` |

名前が単一名（姓が無い、例:「葵」）の場合はローマ字化した1語のみを使う（ディレクトリ名 `Aoi`、スラッグ `aoi`）。
海外名（カタカナの姓名がはっきり分かれない、例:「ソーン・クラインベック」）は自然に分かち書きしてローマ字化する。

`shortId`（3文字の英大文字コード）は以下のルールで機械的に算出する。

1. 名（下の名前）のふりがなをローマ字化する（例: ことこ → `KOTOKO`）。
2. ローマ字を1文字ずつのモーラ（かな1文字に対応する読み）単位に分割する（例: `KOTOKO` → `KO`/`TO`/`KO`、`しらべ`→`SHIRABE` → `SHI`/`RA`/`BE`、`かいと`→`KAITO` → `KA`/`I`/`TO`）。
3. 各モーラの先頭1文字を取り出して連結する（例: `KO,TO,KO`→`KTK`、`SHI,RA,BE`→`SRB`、`KA,I,TO`→`KIT`）。

既存の `shortId`（`src/content/characters/*.json` から収集）と重複していないかを確認し、**重複した場合のみ**ユーザーに代替案を確認する。重複がなければそのまま採用してよい。

## 2. `public/images` ディレクトリの作成（ルール①）

`public/images/{ディレクトリ名}/` を作成し、直下に空の

- `icon/`
- `picture/`
- `sprite/`
- `companions/`

を作成する（4つとも必ず作成する。実ファイルの配置はユーザーが別途行うため空でよい）。

## 3. キャラJSONの作成（ルール②） `src/content/characters/{N}-{スラッグ}.json`

ファイル名の連番 `{N}` は **`src/content/characters.json` 配列の長さ（0始まりのインデックス）と一致させる**（2桁ゼロ埋め。`00`〜）。これはルーティング側 (`src/pages/characters/[name]/index.tsx`) が `${index}-${characterId}` という名前でファイルを解決しているため必須。

### 3-1. フィールド対応表

シートの見出しは以下の想定（`ccfolia`形式のCoC6/7版シート）。

- `基本情報`（名前・職業・年齢・性別・身長・体重・出身・誕生日）
- `アイコン`（画像URL。**これは無視してよい** — thumbnailUrlは後述の規約パスを使う）
- `能力値`（STR/CON/POW/DEX/APP/SIZ/INT/EDU/HP/MP/正気度/IDE/幸運/知識の表。列は「現在値／能力値／増加値／一時的」）
- 能力値表の下の `正気度 x/y`, `DB`, `BLD`, `MOV` の行 → **JSONには保存しない（無視）**
- `技能値`（職業ポイント/趣味ポイントの合計。**無視**）
- `戦闘技能` / `探索技能` / `行動技能` / `交渉技能` / `知識技能`（各技能の「合計／初期値／職業P／趣味P／成長分／その他」の表）
- `武器`, `装備と所持品`, `収入と財産` → **無視**（JSONに対応フィールドなし）
- `フリーテキスト` 配下の `[容姿の特徴]` `[イデオロギー/信念]` `[重要な人々]` `[気に入っている場所]` `[秘蔵の品]` `[特徴]` `[負傷や傷跡]` `[持ち物やお気に入りのアイテム]` `[魔導書や恐怖のアーティファクト]` `[遭遇した超自然の存在]` `[間違ったシナリオ観]`
  - このうち **`[重要な人々]`** に書かれた人物一覧は `histories[].companions` の元ネタになる（3-4参照）。それ以外の空欄の見出しは無視してよい。
- `メモ`（自由記述。末尾にQ&A形式の設問が書かれていることがある）

| JSONパス | 導出元 | 備考 |
|---|---|---|
| `shortId` | 手動確認（1-の通り） | |
| `name` | 基本情報 > 名前 | |
| `enName` | 基本情報 > 名前のふりがな（ローマ字化） | |
| `thumbnailUrl` | 規約パス `/images/{ディレクトリ名}/{ディレクトリ名}_ad.png` | シートの「アイコン」URLは使わない（サイト用に別途変換して置くため） |
| `colorPalette` | シートに情報なし。**ダミーでグレー系3色を配置**（例: `["#CCCCCC", "#CCCCCC", "#CCCCCC"]`）。公開前にユーザーが本来の配色に差し替える | 3色配列 |
| `backgroundUrl` | 規約パス `/images/bg/{ディレクトリ名}_bg.jpg` | |
| `profile.description` | `メモ` セクションの本文をそのまま（改行・空行そのまま）コピー。末尾にQ&Aがあればそのまま含める | |
| `profile.age` | 基本情報 > 年齢（文字列として） | |
| `profile.gender` | 基本情報 > 性別。「女」→「女性」、「男」→「男性」に正規化 | |
| `profile.height` | 基本情報 > 身長 | |
| `profile.weight` | 基本情報 > 体重 | |
| `profile.job` | 基本情報 > 職業 | |
| `profile.birthday` | 基本情報 > 誕生日 | |
| `profile.from` | 基本情報 > 出身。記載があればそのまま使用。**空欄なら `"ダミー"` を入れておく**（本文中に出身地を匂わす記述があっても機械的には補完しない） | |
| `sprites` | プレースホルダーとして2件生成: `sprite/{Dir}_sprite_0.png`+`icon/{Dir}_icon_0.png`、`_1` | 実枚数はユーザーが画像を用意した後に調整 |
| `status.type` | シート冒頭のバージョン表記（「6版」→`"CoC6版"`、「7版」→`"CoC7版"`） | |

| `skills` | 3-2 参照 | |
| `histories` | 3-4 参照（新規キャラは1件のみ） | |
| `qaList` | 3-3 参照（回答はユーザーが自分で入力するため空欄でよい） | |
| `comment` | シートに無い、作者（制作者）自身のメタコメント。**ダミーで `"ダミー"` を入れておく**（公開前にユーザーが本文を執筆） | |
| `artGallery` | `[]`（空配列。ギャラリー画像は後で追加） | |

### 3-2. `skills` の抽出ルール（重要）

各技能テーブルの行は「合計／初期値／職業P／趣味P／成長分／その他」の6列。
**「職業P・趣味P・成長分・その他」のいずれか1つでも0より大きい行だけを採用**し、キーを技能名、値を「合計」列にする。初期値のまま（4列とも0）の行は出力しない。

- 技能名が `A/B(専門分野)` のようなスラッシュ併記（例:「芸術/製作(作曲)」）の場合、実際に選んだ方の表記だけを使う（例: `製作(作曲)`）。すでに `芸術(キーボード)` のように単独で専門分野が書かれている行はそのまま使う。
- 採用順は「戦闘技能 → 探索技能 → 行動技能 → 交渉技能 → 知識技能」の順、各カテゴリ内はシート記載順を維持する（`03-kanade-kotoko.json` はこの順で生成されている）。

### 3-3. `qaList` の生成ルール

サイト側 (`src/components/pc/QAFrame/QAFrame.tsx`) に固定の4問が埋め込まれており、キャラJSON側には回答のみを保持する。**回答の創作はしない**（ユーザーが後で自分の言葉で入力するため）。順序・件数（4件）だけ合わせ、`answer` は空文字にしておく。

1. 休日何してる？
2. 待ち合わせ、早め？ちょうど？遅れがち？
3. もしも願い事が一つ叶うとしたら？
4. もしも明日地球が滅亡するなら何をする？

各要素は `{ "answer": "", "iconUrl": "/images/{Dir}/icon/{Dir}_icon_0.png" }`（iconUrlは作成したプレースホルダーのいずれかを使う）。

### 3-4. `histories` の生成ルール（新規キャラは1件）

- `title`: そのキャラが初登場するシナリオ名。シートには無い情報なので **`"ダミー"` を入れておく**（公開前にユーザーが実際のシナリオ名に差し替える）。
- `comment`: キャラの口調で書かれた短い一言（創作してよい）。
- `iconUrl`: 作成したプレースホルダーのいずれか。
- `companions`: シートの `フリーテキスト > [重要な人々]` に列挙された人物から作る。各要素は
  - `name`: 記載の氏名
  - `enName`: ローマ字化して大文字（1-の要領）
  - `nickName`: シートに書かれた「呼び方」
  - `color`: シートに無い。**ダミーでグレー（例: `"#CCCCCC"`）を入れておく**
  - `iconUrl`: 専用画像が無ければ `"/images/common/companions_NoImage.png"` を使う

## 4. サイトへの登録（画像・JSON作成だけでは画面に出ないため必須）

`public/images` と `src/content/characters/*.json` を作っただけでは、ルーティングが `CharactersId` と `characters.json` を突き合わせて解決するため、**新規キャラはページに表示されない**。以下も同時に追記する。

1. `src/content/characters.json` の配列末尾に1件追加:
   ```json
   {
     "id": "{スラッグ}",
     "name": "{name}",
     "enName": "{enName}",
     "thumbnailUrl": "/images/{Dir}/{Dir}_ad.png",
     "color": "{colorPaletteの1色目}"
   }
   ```
2. `src/definitions/charactersName.ts` の `CharactersId` オブジェクト末尾に1行追加:
   ```ts
   {PascalCaseのスラッグ}: "{スラッグ}",
   ```
   （例: `KanadeKotoko: "kanade-kotoko",`）

これらは既存の並び順（`characters.json` の配列順 = `CharactersId` の定義順 = 詳細JSONのファイル番号）を厳密に一致させて末尾に追加すること。

## 5. 最終確認

- 生成した4ファイル（画像ディレクトリ、詳細JSON、`characters.json`、`charactersName.ts`）の一覧を提示する。
- 以下はシートから読み取れないため **ダミー値／空欄のまま生成する**（ユーザーに質問して手を止めない。生成後に一覧として提示するのみ）:
  - `colorPalette`: グレー3色（ダミー）
  - `profile.from`: シートが空欄だった場合は `"ダミー"`
  - `qaList[].answer`: 4件とも空文字（ユーザーが後で入力）
  - `histories[0].title`: `"ダミー"`
  - `histories[0].companions[].color`: グレー（ダミー）
  - `comment`: `"ダミー"`
- `shortId` は 1- のルールで機械的に算出し、既存と重複した場合のみユーザーに確認する。
- JSONとして正しくパースできるか（`python -m json.tool` 等）を確認する。
- 生成後、上記ダミー値の一覧を箇条書きで提示し、「公開前に手入力で置き換えが必要な項目」として明示する。
